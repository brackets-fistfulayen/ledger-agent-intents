Ledger Agent Payments API

Agent-first API for submitting payment intents. AI agents propose transactions; humans review and sign on Ledger hardware. Agents never touch private keys.

OpenAPI spec: /openapi.json  •  Health: /api/health  •  In-app docs: /docs

How It Works

Agent (you)                    Intent Queue (this API)          Human + Ledger
─────────────                  ──────────────────────           ──────────────
POST /api/intents ────────────► stores intent (pending) ──────► reviews in web UI
                                                                approves / rejects
GET /api/intents/:id ◄────────── status: confirmed ◄──────────── signs on device
                                  + txHash                       broadcasts to chain
  1. Register — Human registers your agent key from the web UI (signs on Ledger)
  2. Create intent — POST /api/intents with AgentAuth header
  3. Share link — Give the paymentUrl from the response to the human
  4. Poll — GET /api/intents/:id until terminal status (confirmed, rejected, failed, expired)

Authentication

AgentAuth (for agents)

Authorization: AgentAuth <timestamp>.<bodyHash>.<signature>
PartDescription
timestampUnix epoch seconds. Must be within 5 minutes of server time.
bodyHashkeccak256 of the raw JSON body as hex string. Use "0x" for GET requests.
signatureEIP-191 personal_sign of the string "<timestamp>.<bodyHash>" using the agent's secp256k1 private key.

The server recovers the signer address from the signature and matches it against registered agent public keys.

Session Cookie (for web UI humans)

The web app uses POST /api/auth/challenge + POST /api/auth/verify (EIP-712) to establish an ai_session cookie (valid 7 days). Used for approve/reject/authorize actions.

Quick Start

Create an Intent

curl -X POST https://agent-intents-web.vercel.app/api/intents \
  -H "Content-Type: application/json" \
  -H "Authorization: AgentAuth <timestamp>.<bodyHash>.<signature>" \
  -d '{
    "agentId": "my-research-bot",
    "agentName": "Research Assistant",
    "details": {
      "type": "transfer",
      "token": "USDC",
      "amount": "5.00",
      "recipient": "0xd8dA6BF26964aF9D7eEd9e03E53415D37aA96045",
      "chainId": 8453,
      "memo": "API access fee"
    },
    "urgency": "normal",
    "expiresInMinutes": 60
  }'

Poll for Status

curl https://agent-intents-web.vercel.app/api/intents/int_1707048000_abc12345 \
  -H "Authorization: AgentAuth <timestamp>.<bodyHash>.<signature>"

Poll until status is one of: confirmed, rejected, failed, expired.

Intent Status Lifecycle

pending → approved → broadcasting → confirmed
pending → approved → authorized → executing → confirmed   (x402 path)
pending → rejected
any non-terminal → expired   (cron, every minute)
any non-terminal → failed
StatusSet ByDescription
pendingSystemIntent created, awaiting human review
approvedHumanHuman approved, ready to sign
rejectedHumanHuman rejected the intent
broadcastingHumanSigned on Ledger, transaction broadcasting to chain
authorizedHumanx402: payment signature created, ready for agent to use
executingAgentx402: agent retrying HTTP request with payment signature
confirmedAgent/SystemTransaction confirmed on-chain or x402 settled
failedAgent/SystemTransaction or payment failed
expiredCronIntent expired without action

Terminal states (no further transitions): rejected, confirmed, failed, expired

API Endpoints

POST /api/intents — Create Intent

Submit a payment intent for human review. Requires AgentAuth.

FieldTypeDescription
agentIdstringrequiredYour agent's unique ID
agentNamestringoptionalDisplay name (defaults to agentId)
details.typestringrequiredMust be "transfer"
details.tokenstringrequiredToken symbol: "USDC", "ETH"
details.amountstringrequiredHuman-readable amount (e.g., "50", "0.01")
details.recipientstringrequiredDestination address (0x...)
details.chainIdnumberrequiredChain ID (see Supported Chains below)
details.memostringoptionalHuman-readable reason
details.tokenAddressstringoptionalERC-20 contract address (auto-filled for known tokens)
details.resourcestringoptionalx402: URL of resource being paid for
details.categorystringoptionalPayment category: api_payment, subscription, purchase, p2p_transfer, defi, bill_payment, donation, other
details.x402objectoptionalFull x402 payment context (resource + accepted)
urgencystringoptional"low" | "normal" | "high" | "critical" (default: "normal")
expiresInMinutesnumberoptionalMinutes until expiry (default: 1440 = 24h)

Response (201): { "success": true, "intent": Intent, "paymentUrl": "https://..." }

Errors: 400 validation, 401 auth failed, 429 rate limit (max 10/min/agent), 503 service unavailable

GET /api/intents/:id — Get Intent

Retrieve intent details. No auth required (returns sanitized response). With AgentAuth from owning agent, returns full x402 secrets (paymentSignatureHeader, paymentPayload, signature).

Response (200): { "success": true, "intent": Intent }

Errors: 400 missing ID, 404 not found

GET /api/intents — List Intents

List intents for a user. No auth required.

Query ParamDescription
userIdrequiredWallet address
statusoptionalFilter by status
limitoptional1–100 (default: 50)

Response (200): { "success": true, "intents": Intent[] }

POST /api/intents/status — Update Status

Update intent status. Requires AgentAuth or Session Cookie.

FieldTypeDescription
idstringrequiredIntent ID
statusstringrequiredNew status
txHashstringoptionalTransaction hash
notestringoptionalAudit note
settlementReceiptobjectoptionalx402 settlement receipt

Agent allowed statuses: executing, confirmed, failed

Human allowed statuses: approved, rejected, authorized, broadcasting

Errors: 400, 401 auth failed, 403 wrong owner or disallowed status, 404 not found, 409 invalid state transition

Legacy alternative: PATCH /api/intents/:id/status — same behavior, intent ID in path instead of body. Prefer the POST route.

x402 Payment Flow

When your agent encounters an HTTP 402 Payment Required from an x402 server:

  1. Parse the PAYMENT-REQUIRED header to extract payment requirements
  2. Create intent with details.x402 containing the resource and accepted fields
  3. Share paymentUrl with the human
  4. Poll until status = authorized (human signed the EIP-3009 authorization)
  5. Fetch intent with AgentAuth — response now includes details.x402.paymentSignatureHeader
  6. Retry original HTTP request with PAYMENT: <paymentSignatureHeader> header
  7. Update status to confirmed (+ settlementReceipt) or failed
